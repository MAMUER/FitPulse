# FitPulse — Техническое задание

> Полное ТЗ. Для краткого обзора см. [README.md](README.md).

## 1. Наименование и область применения

**1.1.** FitPulse — интеллектуальная мобильная веб-платформа для персонализированного управления физической активностью, прогнозирования рисков на основе анализа биометрических данных.

**1.2.** Область применения: автоматизация планирования тренировок, мониторинг состояния здоровья, профилактика перетренированности и травм, генерация диеты.

## 2. Основание для разработки

**2.1.** Учебный проект по дисциплине «Программная инженерия».

**2.2.** Цель: масштабируемая платформа с индивидуальным подбором тренировок и диеты на основе биометрического анализа с носимых устройств.

## 3. Назначение разработки

**3.1.** Для пользователя: регистрация, аутентификация, подключение носимых устройств, сбор биометрии, классификация состояния, генерация тренировок и диеты, адаптивная модификация плана, история и прогресс.

**Для администратора:** регистрация через invite-код, управление пользователями, создание invite-кодов.

**3.2.** Эксплуатация: мобильное веб-приложение (6 вкладок: Обзор, Профиль, Тренировки, Устройства, Диета).

**3.3.** Детали: проверка биометрических данных, многоканальная интеграция (ROOK/Terra/Health Connect/HealthKit), гибридный подход для противопоказаний, криптозащита резервных копий, canary-деплой.

## 4. Требования к программе

### 4.1. Функциональные характеристики

**4.1.1.** Система обеспечивает:

- регистрацию/авторизацию с email-верификацией (SMTP);
- аутентификацию по JWT (ES256, TTL 15 мин) + Refresh Token (opaque, 7 дней, rotation);
- CRUD профиля пользователя;
- CRUD биометрических данных;
- регистрацию и приём данных с носимых устройств;
- классификацию состояния по 6 признакам;
- генерацию тренировочных планов на 4–12 недель;
- генерацию диеты с расчётом макронутриентов;
- ежедневную адаптивную модификацию плана;
- разграничение ролей (client, admin);
- логирование (slog) + OpenTelemetry traces;
- HMAC-SHA256 подпись критических ответов;
- принудительную инвалидацию сессии при logout;
- однократное использование auth-кодов;
- замену 403 на 404;
- маскировку версий ПО;
- строгую CSP nonce-based;
- Rate limiting (per-IP 10r/s, per-user 100r/s, burst 20);
- Argon2id хеширование паролей (128MB+ memory, 3 iters, 4 parallelism).

**4.1.2.** Состав подсистем:

|Подсистема|Назначение|
|---|---|
|User Service|Регистрация, логин, профили, email-верификация, invite-коды|
|Biometric Service|Приём и хранение биометрических данных|
|Training Service|CRUD тренировочных планов, история, достижения|
|Device Connector|Подключение и приём данных с носимых устройств|
|ML Classifier|Классификация состояния (6 классов, Keras, TensorFlow)|
|ML Generator|Генерация планов (GAN)|
|ML Engine|Комплексная логика: диета, адаптивные тренировки|
|Gateway|Единая точка входа (REST → gRPC)|
|NGINX|SSL termination, CSP, rate limiting|

### 4.2. Требования к надёжности

- автоматический перезапуск контейнеров;
- health-check (`GET /health`);
- graceful shutdown;
- обработка ошибок подключения к БД;
- HMAC-SHA256 подпись критических ответов;
- валидация входных данных (gRPC status codes);
- deduplication данных с устройств (device_id + timestamp + metric_type);
- SAST (gosec), govulncheck, Trivy, TruffleHog, Gitleaks в CI/CD.

### 4.3. Условия эксплуатации

- сервер: Linux x86_64, 4 ГБ ОЗУ, 20 ГБ диска;
- клиент: мобильный веб-браузер с поддержкой ES2026;
- сеть: HTTPS (порт 8443), TLS 1.3 only.

### 4.4. Минимальные требования к серверу

- CPU: 2 ядра
- RAM: 4 ГБ
- Диск: 20 ГБ (SSD рекомендуется)
- Сеть: 100 Мбит/с

### 4.5. Информационная совместимость

**Входные данные:** email (RFC 5322, max 254), пароль (min 8), invite-код (`ROLE-YYYY-<hash>`, max 50), биометрия (heart_rate 30–220, spo2 70–100, temperature 35.5–38.5, blood_pressure 80–200/50–130), роль (client, admin), устройство (apple_watch, samsung_galaxy_watch, huawei_watch_d2, amazfit_trex3).

**Выходные данные:** JSON (UTF-8), Protobuf (binary).

### 4.6. Транспортные протоколы

- Клиент ↔ Сервер: HTTPS (TLS 1.3), JSON REST API
- Gateway ↔ Микросервисы: gRPC over HTTP/2
- Сервер ↔ SMTP: SMTP over TLS (порт 465)
- Микросервисы ↔ RabbitMQ: AMQP 0.9.1

## 8. Порядок контроля и приёмки

### 8.1. Контроль качества

- `make check` — fmt + vet + lint + test + build
- `go vet ./...` — статический анализ
- `golangci-lint run` — расширенный линтинг
- `go test -v -timeout 5m ./...` — unit-тесты
- Покрытие ≥ 80% (`go test -coverprofile=coverage.out ./...`)

### 8.2. CI/CD Pipeline

`test` → `security-scan` (gosec, govulncheck, Trivy) → `secrets-scan` (TruffleHog, Gitleaks) → `build` → `docker` → `deploy` → `notify` (Telegram).

### 8.3. Нагрузочное тестирование

- k6 (script: `scripts/load-test/load-test.js`), 50 VU, 2 min.

### 8.4. Критерии приёмки

- все unit-тесты проходят;
- линтер без ошибок (0 issues);
- security-scan без критических уязвимостей;
- приложение разворачивается через `docker compose up -d`;
- регистрация → верификация email → логин → получение профиля работают последовательно.
