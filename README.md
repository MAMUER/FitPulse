# FitPulse — Интеллектуальная платформа персонализированных тренировок

---

## Технические сведения о программе

### 1. Общие сведения

**Полное наименование:** FitPulse — интеллектуальная веб-платформа для персонализированного управления физической активностью, прогнозирования рисков на основе анализа биометрических данных.

**Краткое наименование:** FitPulse.

**Версия программы:** 2.0.0.

**Платформа:**
- Серверная часть: Linux (Docker, Kubernetes), Windows (разработка)
- Клиентская часть: мобильный веб-браузер (Chrome, Firefox, Edge, Safari)

**Языки разработки:** Go 1.26, Python 3.11, JavaScript (ES2022), SQL.

### 2. Функциональное назначение

Программа предназначена для:
- регистрации и авторизации пользователей с подтверждением email (SMTP);
- отдельной регистрации администраторов через **invite-коды**;
- сбора и хранения биометрических данных с носимых устройств (Apple Watch, Samsung Galaxy Watch, Huawei Watch D2, Amazfit T-Rex 3);
- классификации текущего состояния пользователя с помощью ML-модели (6 классов: восстановление, выносливость E1-E2, пороговая E3, силовая/HIIT, перетренированность, заболевание);
- генерации индивидуальных тренировочных планов с использованием GAN;
- генерации рекомендованной диеты с расчётом макронутриентов;
- ежедневной адаптивной модификации плана тренировок при отклонении биометрических показателей;
- управления профилем пользователя (возраст, вес, рост, цели, противопоказания, питание, сон);
- мониторинга состояния здоровья в реальном времени;
- отслеживания достижений и прогресса тренировок.

### 3. Описание программы

FitPulse представляет собой комплексную платформу, которая объединяет данные с носимых устройств, адаптивную ML-логику и персонализированную генерацию тренировок и рациона. Система поддерживает интеграцию через агрегаторы, нормализует биометрические потоки, применяет меры безопасности на уровне аутентификации, подписи ответов и защиты API, а также предоставляет прозрачную архитектуру для масштабирования в Kubernetes.

### 4. Описание логической структуры

Система построена по микросервисной архитектуре с элементами гексагональной (порты и адаптеры) архитектуры на уровне каждого сервиса. Каждый микросервис разделён на слои:

- **Domain** (`internal/domain/`) — бизнес-модели, свободные от зависимостей инфраструктуры.
- **Repository** (`internal/repository/`) — абстрактный слой доступа к данным (PostgreSQL, Redis).
- **Service** (каждый `cmd/*/`) — бизнес-логика, зависящая только от интерфейсов domain и repository.
- **Adapters** — gRPC-обработчики, HTTP-обработчики, ML-клиенты, email-отправители, выступающие «портами» к внешнему миру.

Такой подход обеспечивает тестируемость domain-логики без поднятия БД, простоту замены инфраструктуры (например, PostgreSQL → ClickHouse) и чёткое разделение ответственности.

**Архитектурная документация**: Подробное описание инфраструктуры, наблюдаемости, безопасности и процессов развертывания см. в [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

**Архитектурные решения**: ADR (Architecture Decision Records) доступны в [docs/adr/](docs/adr/).

### 6. Настройка и установка

#### Требования
- Docker 24+ / Docker Compose v2
- Go 1.26+ (для локальной разработки)
- Python 3.11+ (для ML-сервисов)
- 4 ГБ ОЗУ минимум

#### Развёртывание (Kubernetes)

```bash
minikube start
kubectl apply -f configs/k8s/namespace.yaml
kubectl apply -f configs/k8s/configmap.yaml
kubectl apply -f configs/k8s/deployments/
kubectl apply -f configs/k8s/services/
kubectl get pods -n fitness-platform -w
```

#### Настройка SMTP (Yandex)

1. Создать пароль приложения: https://passport.yandex.ru/profile → Безопасность → Пароли приложений
2. В `.env` указать:
   ```
   SMTP_HOST=smtp.yandex.ru
   SMTP_PORT=465
   SMTP_USER=ваш_email@yandex.ru
   SMTP_PASSWORD=пароль_приложения
   SMTP_FROM=ваш_email@yandex.ru
   SMTP_TLS=true
   ```

#### Регистрация первого администратора

```sql
-- Выполнить в БД:
SELECT create_invite_code('admin', NULL, 1, 365);
-- Вернёт код вида: ADMIN-2026-<hash>
```

Затем зарегистрироваться через `POST /api/v1/register/invite` с этим кодом.

### 7. Входные и выходные данные

**Входные данные:**
- Данные пользователя: email, пароль, ФИО, роль (client, admin)
- Invite-код для регистрации админа
- Биометрические данные: тип метрики (heart_rate, spo2, temperature, ecg, blood_pressure, sleep, steps, hrv), значение, timestamp, устройство
- Параметры профиля: возраст, пол, рост, вес, уровень подготовки, цели, противопоказания, питание, сон
- Параметры тренировки: длительность (недели), доступные дни, время суток, место (дом/зал/бассейн/улица)
- Данные здоровья: заболевания, особенности тренировок

**Выходные данные:**
- JWT access token (HS256, TTL 24 ч)
- Тренировочный план (JSON: упражнения, подходы, повторения, длительность, расписание по дням и времени суток)
- Рекомендованная диета (JSON: приёмы пищи, калории, макронутриенты)
- Классификация состояния (JSON: класс, уверенность, рекомендации)
- Письмо верификации email (HTML)
- HMAC-SHA256 подпись критических ответов

## Техническое задание

### 1. Наименование и область применения

**1.1.** Настоящее техническое задание распространяется на программный комплекс «FitPulse» — интеллектуальную мобильную веб-платформу для персонализированного управления физической активностью, прогнозирования рисков на основе анализа биометрических данных.

**1.2.** Область применения: автоматизация планирования тренировок, мониторинг состояния здоровья, профилактика перетренированности и травм, генерация диеты.

### 2. Основание для разработки

**2.1.** Разработка выполнена в рамках учебного проекта по дисциплине «Программная инженерия».

**2.2.** Цель разработки: создание масштабируемой платформы, обеспечивающей индивидуальный подбор тренировочных нагрузок и диеты на основе биометрического анализа с носимых устройств.

### 3. Назначение разработки

**3.1.** Программа предназначена для:

**Для пользователя:**
- регистрации и аутентификации с двухфакторной верификацией email;
- подключения носимых устройств (Apple Watch, Samsung Galaxy Watch, Huawei Watch D2, Amazfit T-Rex 3);
- непрерывного сбора биометрических показателей (ЧСС, ЭКГ, давление, сатурация, температура, сон, шаги, HRV);
- автоматической классификации текущего состояния пользователя (6 классов);
- генерации адаптивных тренировочных планов с применением GAN (расписание по дням, упражнениям, времени суток);
- генерации рекомендованной диеты с расчётом макронутриентов;
- ежедневной адаптивной модификации плана при отклонении показателей;
- ведения истории тренировок и отслеживания прогресса;
- уведомлений о рисках и рекомендациях.

**Для администратора:**
- регистрации через invite-код;
- управления пользователями;
- создания invite-кодов для новых администраторов.

**3.2.** Эксплуатация: мобильное веб-приложение, доступное через браузер (6 вкладок: Обзор, Профиль, Тренировки, Устройства, Диета).

**3.3.** Техническое задание определяет ключевые требования по безопасности, интеграции с носимыми устройствами, адаптивности ML, управлению пользователями и надежности. В документе подробно прописаны:
- проверка и нормализация биометрических данных;
- поддержка многоканальной интеграции через ROOK/Terra/Health Connect/HealthKit;
- гибридный подход для противопоказаний: структурированные коды + текстовые примечания;
- криптозащита резервных копий и восстановление;
- канаре-стратегия развертывания и автоматический откат.

### 4. Требования к программе и программному документу

#### 4.1. Требования к функциональным характеристикам

**4.1.1.** Система должна обеспечивать:
- регистрацию пользователей с подтверждением email (SMTP);
- отдельную регистрацию администраторов через invite-коды;
- аутентификацию по JWT (HS256, TTL 24 ч);
- CRUD-операции над профилем пользователя;
- добавление и получение биометрических данных;
- регистрацию и приём данных с носимых устройств;
- классификацию состояния по 6 признакам;
- генерацию тренировочных планов на 4–12 недель (с расписанием по дням и времени суток);
- генерацию диеты с расчётом макронутриентов;
- ежедневную адаптивную модификацию плана тренировок;
- разграничение ролей (client, admin);
- логирование всех операций (zap, уровень info/warn/error);
- HMAC-SHA256 подпись критических ответов;
- принудительную инвалидацию сессии при logout;
- однократное использование auth-кодов;
- разделение хранилищ сессий для критических действий;
- замену кода 403 на 404 Not Found;
- маскировку версий серверного ПО;
- кастомизацию страниц ошибок;
- немедленное завершение выполнения после отправки управляющих заголовков;
- использование промежуточных слоёв (middleware);
- серверную валидацию привилегий через прямой запрос к БД;
- строгую политику Content Security Policy.

**4.1.2.** Состав подсистем:

| Подсистема | Назначение |
|------------|------------|
| User Service | Регистрация, логин, профили, email-верификация, invite-коды |

#### 4.2. Требования к надёжности

**4.2.1.** Система должна обеспечивать:
- автоматический перезапуск контейнеров при сбое (`restart: unless-stopped`);
- health-check для PostgreSQL и Redis;
- graceful shutdown всех сервисов;
- обработку ошибок подключения к БД (логирование, возврат 500);
- HMAC-SHA256 подпись критических ответов (login, register, профиль, биометрия, планы);
- валидацию всех входных данных (gRPC status codes);
- deduplication данных с устройств (device_id + timestamp + metric_type);

- сканирование Docker образов (Trivy) в CI/CD;

## CI/CD Pipeline

```
push/PR → test → security-scan → secrets-scan → build → docker → deploy → notify
```

- **test**: go vet, golangci-lint, go test
- **security-scan**: gosec (SAST), govulncheck, Trivy
- **secrets-scan**: TruffleHog, Gitleaks
- **build**: компиляция Go бинарников
- **docker**: сборка и push Docker образов
- **deploy**: VPS (Docker Compose) + Kubernetes
- **notify**: Telegram с полным отчётом

### Kubernetes (опционально / на перспективу)

Job `Deploy to Kubernetes` в CI/CD настроен как **опциональный**: если GitHub Secret `KUBECONFIG` пустой или отсутствует, шаг gracefully пропускается без ошибки.

**Почему K8s не активен по умолчанию:**
- Managed Kubernetes (Yandex Cloud, DO, AWS EKS, GKE) — платные сервисы, требуют отдельного бюджета.
- Самостоятельная установка кластера через kubeadm/k3s/minikube на одном VPS неоправданно сложна и не даёт реальных преимуществ по сравнению с Docker Compose.
- Текущий деплой полностью покрывает потребности через Docker Compose на VPS.

**Для активации K8s-деплоя:**
1. Создать кластер в облаке (Managed K8s).
2. Скачать `kubeconfig` с панели управления.
3. Сохранить содержимое YAML (или его base64-кодировку) в GitHub Secret `KUBECONFIG`.
4. Тогда CI автоматически применит манифесты из `configs/k8s/`.

## Сообщить об ошибке

Мы внимательно относимся к bug-репортам и улучшаем проект с помощью сообщества.

### Как correctly заполнить issue

1. **Проверьте, не заведён ли баг ранее** — поищите в [Issues](https://github.com/MAMUER/Project/issues).
2. **Используйте шаблон** — при создании issue выберите шаблон «Bug report».
3. **Прикрепите логи** — если проблема воспроизводится локально, добавьте фрагменты логов из:
   - `docker compose logs <service>` для Docker-окружения
   - `journalctl` / логи Kubernetes-поднятого кластера
4. **Укажите окружение** — версия Docker/Compose, ОС, браузер, если имеет значение.
5. **Для критических уязвимостей** — откройте issue с меткой `security` и опишите проблему максимально кратко (не раскрывайте PoC). Мы свяжемся с вами для дополнительных деталей.
